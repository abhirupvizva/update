Service Structure – Final Approach Proposed Implementation

How It Works

Inside the Deal layout, we create predefined service sections:

  • Service Block 1
    • Category 1 (Dropdown)
    • Subcategory 1 (Dependent Dropdown)
    • Base Service 1 (Dependent Dropdown)
    • Flavor 1 (Dependent Dropdown)
  • Service Block 2
    • Category 2
    • Subcategory 2
    • Base Service 2
    • Flavor 2
  • Continue up to a fixed number (e.g., 6 or 9).
  • Each block has its own dependency mapping.

Why This Is Fully Doable

  • Freshsales supports multiple dropdown fields
  • Field dependencies per field
  • Conditional visibility
  • Required field rules per block
  • There is no automation required. Everything is native configuration.

How To Keep It Clean

  • Show Service Block 2 only if Service Block 1 Category is selected.
  • Show Service Block 3 only if Block 2 is used.
  • This prevents layout clutter.

Pros

  • 100% dropdown based
  • Same cascade logic repeated
  • Very stable
  • Clean reporting (each block is a fixed field)
  • No API complexity

Limitations

  • Maximum number must be predefined.
  • If one deal needs 12 services but we built 6, expansion is required.

Product Catalog Pricing Decision Pending

Since dynamic pricing is no longer required and users will manually enter Deal Value, the Product Catalog can be removed. However, this should only be done after confirming that no integrations or reports depend on it.

Account Field Clarification Requires Confirmation

The correct native Account lookup must be validated to avoid duplicate account references.

Deal ID Auto-Population Platform Limitation

Webhooks and workflows were tested, but Freshsales does not expose the generated Deal ID during the same save transaction. This cannot be automated at this time.

Overall Direction

  • Dropdown-only service structure is achievable.
  • Nested dynamic UI grouping is not supported.
  • Subform or Custom Module are the two realistic approaches.
