Implementation Summary and Completion Confirmation

This document outlines what has been implemented based on prior discussions and confirms completion of all agreed items.

1 Cascade Structure (Deal Module)

We have finalized the cascading structure inside the Deal module exactly as discussed.

The flow now works as:

  • Category → Subcategory → Base Service → Flavor / Enhancement
  • Each level properly controls the next level.
  • The dependency chain ends at Flavor / Enhancement as a multi-select field.
  • There are no duplicate service groups anymore.
  • A la carte, add-ons, and other services are now merged into the main category hierarchy.
  • All legacy and redundant structures have been removed.

Freshsales dependency constraints addressed:

A dependent field can only have one controlling field. Multi-select cannot act as a controlling field, so the cascade correctly ends at Flavor / Enhancement.

2 Custom Enhancement Logic

We’ve implemented a clean and reliable custom handling structure.

  • Flavor / Enhancement is a multi-select field.
  • A radio button field exists: Custom Enhancement? (Yes / No).
  • If Yes is selected, a Custom Enhancement Text field becomes available.

This approach is more stable than attempting to detect custom entries within multi-select logic, and it provides stronger reporting control.

We can now easily generate reports on:

  • Deals where Custom Enhancement = Yes
  • All custom entries for operational review

3 Product Catalog + Plan B Structure

We followed Plan B exactly as agreed:

  • Product Catalog acts as the master list.
  • Services are maintained there.
  • Active/inactive status is maintained.
  • Placeholder pricing is stored.
  • The Deal module handles guided selection through field dependencies.

We are not forcing configurator logic inside the Product Catalog, since Freshsales is not designed to behave as a full CPQ or product configurator.

This provides:

  • Better usability
  • Cleaner admin control
  • More flexibility in Deals

4 Pricing Structure Resolution

We addressed the pricing limitation discussed.

Because Freshsales locks Deal Value when products are added, we separated pricing into two clear fields.

Implemented fields:

  • Estimated Deal Value (used for system or internal reference)
  • Proposed Amount (client-facing commercial value)

This enables you to:

  • Adjust pricing for negotiation
  • Apply discounts
  • Maintain margin strategy
  • Avoid being restricted by product auto-calculation

This ensures sales flexibility without modifying master product pricing.
